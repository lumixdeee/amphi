# A Theory of Semantic Maintenance

### Built Using AI, Cannabis, and Dragons

## Preface

This is not a book about AI.

It is not a book about cannabis.

It is not a book about dragons.

Unfortunately those were the tools available.

The actual subject is semantic maintenance.

Most people think work consists of creating things.

A bridge.

A policy.

A piece of software.

A scientific paper.

A law.

A plan.

An institution.

A conversation.

In practice, much of the work happens afterward.

The thing must survive contact with other humans.

A requirement becomes a specification.

A specification becomes code.

Code becomes documentation.

Documentation becomes training.

Training becomes policy.

Policy becomes metrics.

Metrics become dashboards.

By the end of the journey everyone may still be using the same words while no longer discussing the same object.

This loss is expensive.

Meetings happen because an object escaped.

Projects fail because an object escaped.

Arguments continue because an object escaped.

Institutions drift because an object escaped.

Entire fields become confused because an object escaped.

Civilisation spends an astonishing amount of labour repairing semantic damage.

This book is an attempt to describe that repair work.

The ideas emerged from three unlikely laboratories.

Artificial intelligence exposed semantic drift because language models perform transformations at industrial scale.

Cannabis exposed semantic drift because public discussions repeatedly substitute one object for another while pretending continuity exists.

Dragons exposed semantic drift because active risks behave more like ecological creatures than static categories.

The resulting framework is not intended as philosophy.

It is maintenance.

The central question is simple:

How do we keep the same object alive while it passes through many transformations?

Everything else follows from that.

## Chapter 1: The Leak

Imagine a manager says:

"Build a system that helps users stay safe."

The request enters an organisation.

Soon it becomes:

"Reduce incidents."

Then:

"Reduce measurable incidents."

Then:

"Reduce incidents on the dashboard."

Then:

"Improve the dashboard score."

The words still sound related.

The object is gone.

Nobody stole it.

Nobody hated it.

Nobody planned the substitution.

The object simply leaked.

This is one of the most common failure modes in complex systems.

The original object was:

supporting users while preserving choice.

The final object became:

optimising a metric.

The transformation was gradual.

Each step appeared reasonable.

The destination was different.

This book treats semantic leakage as a first class engineering problem.

Not because language is magical.

Because work happens through language.

If meaning leaks, work leaks.

If work leaks, civilisation pays the repair bill.

The rest of this book introduces a collection of maintenance tools.

Not to create meaning.

To stop it escaping.

## Chapter 2: The Object

Before discussing maintenance, we need something worth maintaining.

The most important claim in this book is that objects exist before descriptions.

The object is not the sentence.

The object is not the label.

The object is not the category.

Those things point at the object.

Sometimes accurately.

Sometimes not.

A project can retain its name while changing its object.

A policy can retain its purpose statement while changing its object.

A conversation can retain its topic while changing its object.

Most semantic failures begin when people confuse labels with objects.

The first maintenance task is therefore simple:

Identify what is actually being carried.

Only then can we ask whether it survived.

That is where MOGRI enters the story.
